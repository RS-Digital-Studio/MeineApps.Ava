## Critical (Cheat, Datenverlust, Account-Takeover)

### C-01 Co-op-Reward-Doppelauszahlung beim Geraetewechsel
**Datei:** `Services/GuildCoopOrderService.cs:159-178`

`TryClaimCompletedReward` markiert die Order ueber `GameState.ClaimedCoopOrderIds` als
ausgezahlt. `ClaimedCoopOrderIds` lebt im lokalen `GameState` und wird per CloudSave/Lokal
gespeichert. Szenario:

1. Spieler bekommt 50k Reward, `ClaimedCoopOrderIds.Add(OrderId)` lokal.
2. Spieler installiert App neu, laedt aelteres CloudSave (vor der Order).
3. `ClaimedCoopOrderIds` enthaelt die Order-ID nicht mehr → naechster `GetOpenForPlayerAsync`
   ruft `TryClaimCompletedReward` erneut auf → zweite Auszahlung.

Anti-Cheat findet das nicht, weil das HMAC nur `OrderId|CreatedBy|InvitedPlayer|BaseReward|MiniGameType`
abdeckt (Zeile 240). Status/Scores sind nicht im HMAC und der Server pruefte
"already paid" nirgendwo. Mit ueberholtem CloudSave laesst sich der gleiche Order beliebig
oft re-claimen, bis der CoopOrder-Knoten in Firebase manuell geloescht ist.

Mitigation-Idee: Server-Rule `donations`-Style — "already paid"-Marker im CoopOrder-Knoten
mit Player-ID setzen, Rule schreibt-und-validate `claimedBy/<playerId>=true` nur einmal.

### C-02 Cloud-Save ueberschreibt LOKAL haerteren Save ohne Konflikt-Erkennung
**Datei:** `Services/CloudSaveService.cs:105-124, 63-102` + `ViewModels/SettingsViewModel.cs:574+`

`SaveToCloudAsync` ruft `UploadAsync(state)` direkt, ohne den Server-Stand zu lesen oder
mit dem lokalen Stand zu vergleichen. `RestoreFromCloudAsync` analog: schreibt den Cloud-State
ueber `_saveGameService.ImportSaveAsync(cloudJson)` ohne Versions-Pruefung gegen den lokalen
Save. CLAUDE.md verlangt "Cloud-Save hoehere Version → Alert" — der Alert fehlt komplett.

Folge: Spieler hat zwei Geraete, A laeuft auf Level 50, B auf Level 5. B druckt versehentlich
"In Cloud sichern" → Level-50-Save in Firebase wird durch Level-5-Save ueberschrieben.
Datenverlust ohne Recovery-Pfad (kein Versions-Diff/Snapshot/Backup).

Es gibt zwar `GetMetadataAsync` (Zeile 46), aber **keine Aufrufer im SettingsViewModel-Code**
(`grep ".GetMetadataAsync("` liefert nichts). Die `metadata.level/version`-Felder sind
ungenutzte tote Logik.

### C-03 Auction-Bid: kein atomares PATCH → klassische Bid-Race
**Datei:** `Services/WorkerAuctionService.cs:50-84`

`PlaceBidAsync` macht GET-modify-PUT: liest `CurrentAuction` (kein Re-Read von Firebase),
patcht lokales State-Objekt und schreibt das ganze Objekt mit `_firebase.SetAsync(path, auction)`.
Das ist PUT (Last-Write-Wins) — wenn zwei Spieler in derselben 200ms-Lese-Phase die gleiche
`HighestBid` sehen und beide 110% bieten, ueberschreibt der zweite den ersten. Der Verlierer
sieht seinen `myPrevious` korrekt (lokal), aber Firebase verliert sein `AllBids[playerId]=Value`.
Sein Geld ist via `TrySpendMoney(delta)` bereits abgezogen → Verlierer bezahlt, ohne dass
sein Bid jemals in Firebase landet → Refund schlaegt fehl (`ApplyRefunds` sieht ihn nicht in
`auction.AllBids`).

GuildCoopOrderService.cs:103+ hat das (laut Kommentar) genau deshalb auf PATCH umgestellt.
Im WorkerAuctionService nicht. Die 1s-Spam-Cooldown reduziert nur die Wahrscheinlichkeit,
nicht das Risiko bei realer Mehrspieler-Auktion.

### C-04 Coop-Order-Rules erlauben Score-Tampering durch DRITTE
**Datei:** `database.rules.json:36-72` (Root) bzw. `src/Apps/.../database.rules.json:36-65`

Die `.write`-Regel fuer `guilds/$guildId/coopOrders/$orderId` ist:

    "auth != null && (root.child('guild_members/' + $guildId + '/' + root.child('auth_to_player/' + auth.uid).val()).exists())"

Damit darf **jedes Gildenmitglied** den Score, Status, BaseReward, RewardSplit usw. eines
fremden Auftrags ueberschreiben. Es gibt keine Pruefung "auth muss createdBy oder invitedPlayer
sein". Der HMAC im Client (`GuildCoopOrderService.ComputeHmac`) deckt nur stabile Felder ab
(siehe Zeile 234-244) — Score-Tampering bemerkt der Client zwar via `GetStateAsync`-HMAC-Check
(Zeile 187-193), aber ein boeser Spieler kann z.B. `status=Completed` setzen und beide Scores
auf 100 → das Opfer (zweiter Player) sieht beim Polling `Completed + bothPerfect=true` und
`TryClaimCompletedReward` zahlt ihm 125% des Rewards aus, OHNE dass das MiniGame je gespielt
wurde. Mit dem dritten Boes-Player-Konto laesst sich so beliebig Geld in eine Zielgilde
schaufeln.

Mitigation: `.write`-Rule muss `auth.uid in [createdBy, invitedPlayer]` validieren via
`auth_to_player`-Mapping.

### C-05 Auctions-Rules erlauben fremde Bids zu manipulieren
**Datei:** `database.rules.json:74-105` (Root)

`auctions/$auctionId/.write` ist ebenfalls "irgendein Gildenmitglied". Boeser Spieler kann
in fremder Auktion `highestBidderId = his_own_id` setzen, `highestBid = 1` (waere `>= 0`
gemaess validate) und sich selbst als Gewinner einsetzen. HMAC im Client (Zeile 372)
faengt das nur ab, weil `WorkerAuctionService.RefreshAuctionAsync` HMAC re-prueft
(Zeile 103) — aber `ApplyRefunds` (Zeile 184) wird in `SettleAsync` **bevor** HMAC-Re-Check
aufgerufen → der Settler (Master-Client) wendet den manipulierten State an. Anschliessend
schreibt er das settled Result selbst, und alle anderen Clients sehen HMAC-Mismatch und
ignorieren — der Settler hat aber bereits Geld an den falschen Gewinner ueberwiesen
(naja: in WorkerAuctionService.SettleAsync wird kein Geld ueberwiesen — `ApplyRefunds`
gibt es lokal aus, also nur der Settler bezahlt sich selbst — siehe trotzdem unten).

Sekundaerer Fehler in derselben Datei: Validate verlangt `auctionId` als Pflichtfeld,
aber der `endsAt`-Feld-Validator (Zeile 87-89 nur in Root-Rules, in App-Rules komplett
fehlend) erlaubt **String beliebigen Werts** ohne Monotonie. Boes-Spieler kann `endsAt`
verlaengern → eigene Auktion endet nie → andere Bieter blockieren ihr Geld unbegrenzt.

### C-06 Guild-Hall-Level-Cap 50 — aber Skip-Validations bei Top-Level
**Datei:** `database.rules.json:314-336` (Root)

`guild_hall/$guildId/buildings/$buildingId/level` validate ist `>= 0 && <= 50`, aber
**keine Monotonie** (Vergleich mit `data.val()`) und **kein Up-Cost-Check**. Ein Mitglied
kann seinen Gilden-Hall sofort von Level 0 auf 50 fuer ALLE Buildings hochsetzen — es gibt
in den Rules keine Validierung, dass `upgradingUntil` vorher gesetzt war oder dass die
Gilde genug Wochen-Progress hat. Das ist ein guildwide Cheat (Boni gelten fuer alle
Mitglieder).

### C-07 CoopOrder schreibt MiniGameType nicht in Rules-Validate
**Datei:** `src/Apps/.../database.rules.json:36-65` (App-Variante)

In der App-lokalen `database.rules.json` fehlt der `miniGameType`-Validator KOMPLETT (Root
hat ihn auf Zeile 53-55). `expiresAt` fehlt ebenfalls im App-Rules-File. Wenn das die deployed
Variante ist, kann ein Angreifer `miniGameType = 999` oder `expiresAt = "9999-01-01"` setzen
und damit das Polling der Gegenseite einfrieren (Client parst und gibt unverhofftes
Verhalten — z.B. Pending-Order, die nie expiriert, blockiert die `GetOpenForPlayerAsync`-UI-Liste).

## High (Reproduzierbare Bugs, ungenutzte Security-Rule)

### H-01 Anonymer Account-Reset → Verlust der auth_to_player-Bindung
**Datei:** `Services/FirebaseService.cs:120-134`

Wenn Token-Refresh dauerhaft fehlschlaegt, ruft der Code `SignUpAnonymouslyAsync()` und
erzeugt eine **neue Firebase-UID**. Die `PlayerId` bleibt stabil (gut), aber das
`auth_to_player/{oldUid}`-Mapping bleibt in Firebase haengen (NICHT geloescht), und das
neue Mapping wird ueberschrieben. Folge: Bei spaeterer 401-Recovery oder bei einem
Bug der den alten UID aus den Preferences laedt, sind ZWEI gueltige UIDs auf dieselbe
PlayerId gemappt → alle `auth_to_player`-basierten Schreib-Rules erlauben beide Konten
schreibend auf die Gilde. Mitigation: alten Eintrag in `auth_to_player/{oldUid}` vor
dem Signup loeschen (geht nur, wenn der alte ID-Token noch gueltig ist — Henne/Ei).

### H-02 IsOnline-Flag wird im 401-Recovery unkorrekt auf true gesetzt
**Datei:** `Services/FirebaseService.cs:253, 261, 294, 302, 332, 340, 369, 376, 405, 412, 439, 447`

`GetAsync/SetAsync/UpdateAsync/PushAsync/DeleteAsync/QueryAsync` setzen `IsOnline=true`
**sofort** nach `response = await _httpClient.SendAsync(...)`, OHNE den Status-Code zu
pruefen. Eine 401-Antwort (Token expired) oder 500-Antwort beweist nur, dass Firebase
*antwortet*, nicht dass die Operation erfolgreich war. CloudSave/GuildService verlassen
sich auf `IsOnline` um zu entscheiden, ob "Spieler ist definitiv nicht in einer Gilde"
(`GuildService.InitializeAsync:93, 101`). Wenn Firebase 401 zurueckgibt, sieht der Client
"Online + GET liefert null" → `ClearLocalCache()` → loescht Gilden-Membership lokal,
obwohl der Spieler real noch in der Gilde ist. CLAUDE.md erwaehnt diesen Bug explizit als
"IsOnline-Flag korrekt gesetzt in Edge-Cases?" — Antwort: nein.

### H-03 Token-Cooldown setzt fakes `_idToken` "expired_cooldown" als gueltigen Wert
**Datei:** `Services/FirebaseService.cs:486-492`

`ForceRefreshAndRetryAuth` setzt bei dauerhaftem Auth-Fehler:

    _idToken = "expired_cooldown";
    _tokenExpiry = DateTime.UtcNow.AddMinutes(5);

Der Wert `expired_cooldown` ist **kein gueltiger Token**, wird aber an Firebase als
`?auth=expired_cooldown` angehaengt → Firebase antwortet 401. Der Recovery-Pfad in
GET/SET/UPDATE prueft auf 401 und ruft erneut `ForceRefreshAndRetryAuth` → Infinite-Loop
bzw. ein Request macht permanent zwei 401-Roundtrips zu Firebase. Bei 5min Cooldown frisst
das Bandbreite und Quota.

Korrekt waere: `_idToken = null` und in den GET/SET-Methoden auf `_idToken == null` checken,
nicht stur 401-detect.

### H-04 Co-op-MiniGame-Type wird beim Senden NICHT vom Auftraggeber gesetzt
**Datei:** `Services/GuildCoopOrderService.cs:55-61`

`CreateInviteAsync` setzt:

    MiniGameType = Models.Enums.MiniGameType.Sawing,  // Default

Aus dem Kommentar: "das ViewModel waehlt vor dem Aufruf passend". Nirgends im Code wird
`MiniGameType` nach `CreateInviteAsync` ueberschrieben — der ViewModel-Pfad fehlt offenbar
(Phase-B-Status laut Klassen-Kommentar "Service-Logik nur"). Effekt: Jeder Co-op-Auftrag
ist immer SawingGame, egal welcher Workshop einlud. Falls das Feature live geht, ist das
ein hartes UX-Problem (immer dasselbe Spiel), kein Cheat — aber dokumentationswert.

### H-05 GameIntegrityService HMAC ueber FixedTimeEquals mit UTF8-Bytes statt Hex-Vergleich
**Datei:** `Services/GameIntegrityService.cs:60-63`

`CryptographicOperations.FixedTimeEquals` ist timing-sicher — gut. Aber die Bytes werden
aus zwei **HexStringLower-Repraesentationen** generiert (Zeile 71 und 88 nutzen
`Convert.ToHexStringLower`). Das ist semantisch korrekt, aber: falls je einmal eine andere
Funktion `Convert.ToHexString` (uppercase) oder ein Base64-Encoding produziert (z.B.
Migration auf StringComparison.Ordinal), schlaegt der Vergleich falsch fehl — und Spieler
verlieren ihre Save-Sicherheit. Robuster: HMAC-Bytes ROH vergleichen statt Hex-Strings.

### H-06 `MigrateFromUidToPlayerIdAsync` schreibt PlayerName/Role nicht atomar
**Datei:** `Services/GuildService.cs:121-172`

Migration: alter Firebase-UID-Member-Eintrag → neuer PlayerId-Member-Eintrag.
1. `SetAsync(guild_members/{guildId}/{playerId}, memberData)` (Zeile 142)
2. `DeleteAsync(guild_members/{guildId}/{oldUid})` (Zeile 149, mit 3x Retry)

Wenn Delete dauerhaft fehlschlaegt (Network/Permission), bleibt der alte Member als
Geister-Eintrag in der Gilde, **er zaehlt in `memberCount`**, kicked die echte Liste
voller, und CountAndSyncMemberCountAsync syncs ihn weiter. Es gibt keinen Tombstone und
keinen periodischen Cleanup. Die in `RefreshGuildDetailsAsync` (Zeile 703-742) eingebaute
"Duplikat-Erkennung per Name" rettet die UI, aber nicht den `memberCount`.

### H-07 GuildResearchService — kein PerGuild-Cache-Invalidation bei Gilden-Wechsel
**Datei:** `Services/GuildResearchService.cs:17, 404-433`

`_cachedEffects` ist Instance-State. `RefreshEffectsCoreAsync(guildId)` schreibt
`_cachedEffects` immer auf das Ergebnis der zuletzt aufgerufenen Gilde. Szenario: Spieler
verlaesst Gilde A (mit Research X), tritt Gilde B bei (ohne Research X). Wenn zwischen
`LeaveGuild` und dem ersten `GetGuildResearchAsync` der GameLoop einmal
`GetCachedEffects()` ruft (z.B. fuer Crafting-Speed), bekommt er weiter den Effekt aus
Gilde A. `ClearLocalCache()` in GuildService.cs:1278 nullt `GuildMembership` aber **nicht
den GuildResearchService-Cache**.

### H-08 GuildBossService cached `_lastBossCheck` global → blockiert nach Gilden-Wechsel
**Datei:** `Services/GuildBossService.cs:28, 222-225`

`_lastBossCheck = DateTime.UtcNow` ist Instance-Field. Nach `LeaveGuild + Join other Guild`
bleibt `_lastBossCheck` 30 Sekunden lang "vergeben", obwohl der Boss-Knoten der neuen
Gilde NIE gecheckt wurde. Spieler sieht 30s lang einen veralteten Boss-Status (zumindest
keine Schadens-Verteilung), bis der Cooldown abgelaeuft. Bug-Risiko mittel — Reproduzierbar.

### H-09 Cloud-Save `metadata.savedAt` ohne Monotonie-Validierung
**Datei:** `database.rules.json:393-404` (Root) bzw. App-Variante

`cloud_saves/$playerId/metadata.savedAt.validate` ist `isString && length <= 40`. Es gibt
keine Pruefung "neu > alt". Boes-Spieler (oder buggy App) kann `savedAt = "1970-01-01"`
schreiben — naechster Download zeigt "Cloud ist aelter als lokal" obwohl das Cloud-Save
real der neueste ist. Da der Konflikt-Pfad (C-02) ohnehin fehlt, ist das aktuell folgenlos —
aber sobald der Konflikt-Pfad eingezogen wird, ist die Anti-Cheat-Luecke offen.

Aehnlich fehlt fuer `metadata.level`: keine Monotonie (Level kann zurueck auf 0). Bei
Cheatern egal, bei normalen Spielern aber: Soft-Reset/Prestige resettet Level auf 0 — also
**KEINE strenge Monotonie moeglich**. Trotzdem: ein `version`-Feld mit `newData.val() >= data.val()`
fehlt — das schuetzt gegen Cloud-Save-Rollback durch CompromisedClient.

### H-10 SpamBid-Cooldown ist client-side — Server-Rule fehlt
**Datei:** `Services/WorkerAuctionService.cs:58` + Rules

`_lastBidAt` ist Instance-Memory im Client. Ein gerooteter Spieler kann den Service umgehen
und Firebase direkt mit 100 Bids/Sekunde fluten. Rules haben keine `now - data.child('lastBidAt')`-
Check. Wirkung: DoS gegen die eigene Gilde (jeder andere Bid wird vom Spam ueberholt).

### H-11 InviteCode-Kollision: 5 Versuche, dann stiller Re-Use
**Datei:** `Services/GuildInviteService.cs:67-79`

Code-Generator versucht 5x neue Codes; danach wird der zuletzt generierte (kollidierende!)
Code stillschweigend uebernommen und ueberschreibt das `invite_code_to_guild/{code}`-Mapping.
Wenn Gilde A Code `ABC123` hatte und Gilde B den jetzt bekommt, beitreten neue Spieler via
`ABC123` ploetzlich Gilde B (`JoinByInviteCodeAsync` liest `invite_code_to_guild/ABC123`).
Wahrscheinlichkeit gering (36^6 = 2.1Mrd), aber kein Fehlerpfad → keinen `null`-Return
fuer den User.

### H-12 NPC-Bot generiert mit `new Random()` — niedrige Entropie + RNG-Bias
**Datei:** `Services/WorkerAuctionService.cs:291, 326`

`SpawnAuctionIfMasterAsync` und `RunNpcBotTickAsync` instanziieren `new Random()` *pro
Aufruf*. Auf .NET 6+ ist das zwar nicht mehr Time-Based seeded, aber `Random.Shared`
ist thread-safe und teure Realloc spart. Wichtiger: `RunNpcBotTickAsync` wird vom
GuildTickService periodisch gefeuert — bei Master-Client mit z.B. 1s-Tick laufen zwei
Bots im selben Sekundenbruchteil mit moeglicherweise identischem `Random()`-Output
(theoretisch). Cosmetic-Bug, aber `botIndex = rng.Next(0, botCount)` deterministisch
zu wuerfeln ist potenziell unfair.

### H-13 Master-Client wird via `auth_to_player`-Mapping bestimmt aber Members lassen sich faelschen
**Datei:** `Services/WorkerAuctionService.cs:259-280`

`IsMasterClientAsync` liest `guild_members/{guildId}` und nimmt die lex. kleinste Key.
Wenn ein Spieler einen "Geister-Member" (siehe H-06) mit einer Key wie `0000...` einfuegen
kann (Schreibrechte hat er als Gildenmitglied), ueberstimmt er das Master-Voting fuer ALLE
echten Spieler. Geister-Spieler ist `null` ID, Master-Roundup faellt zurueck — der Code
faellt aber bei Exception auf `false` (Zeile 277-279) → KEIN Master → Auktionen werden nie
gespawnt fuer die ganze Gilde. Effektiver Denial-of-Service.

## Medium

### M-01 PipePuzzle: Random-Backtracking-Pfadgenerierung hat keinen Min-Path-Constraint
**Datei:** `ViewModels/MiniGames/PipePuzzleViewModel.cs:200-297`

`FindPath` priorisiert rechts mit ~33% verticalFirst und 1/8 Chance auf links-Schritt. Der
generierte "Loesungs-Pfad" hat keine Mindestlaenge — bei Easy kann der Pfad direkt
Source→Drain ohne Umweg sein, was schwer ist (Spieler haben sehr wenige Drehungen). Bei
Expert dagegen kann er sehr lang sein → Spieler sieht 90% Loesung, aber findet nicht
heraus, dass eine Ecke noch zu drehen ist. Es gibt keine "Optimal-Move"-Berechnung
(CLAUDE.md erwaehnt den Bug) — die Rating-Logik basiert wohl auf `MovesCount` vs.
TimeRemaining, aber Optimal-Moves werden nicht gegen die Path-Laenge geprueft.

### M-02 BaseMiniGameViewModel: GameRestarted-Event wird VOR StartGameAsync gefeuert
**Datei:** `ViewModels/MiniGames/BaseMiniGameViewModel.cs:350-353`

Reihenfolge:
1. `GameRestarted?.Invoke(this, EventArgs.Empty)` (Zeile 350) → View startet Render-Loop
2. `if (!ShowTutorial && !CanAutoComplete) StartGameAsync().SafeFireAndForget()` (Zeile 352-353)

Wenn Tutorial gezeigt wird, hat die View den Render-Loop gestartet und sieht den frueheren
State, bis der Spieler das Tutorial dismissed. Optisch unschoen, aber kein Bug. ABER:
Wenn die View den Tick-Handler in `OnDataContextChanged` setzt und der DataContext nach
`SetOrderId` ein zweites Mal feuert (kann bei View-Pooling passieren), wird der Render-Loop
zweimal gestartet — kein expliziter `_isLooping`-Guard sichtbar.

### M-03 `App.Services` Service-Locator-Antipattern in MiniGames
**Datei:** `ViewModels/MiniGames/BaseMiniGameViewModel.cs:466`

    _analyticsService ??= App.Services?.GetService(typeof(IAnalyticsService)) as IAnalyticsService;

Service-Locator-Antipattern. Falls `App.Services` waehrend View-Pooling oder im Unit-Test
null ist, faellt die Telemetrie aus und der `catch { /* darf nicht crashen */ }` schluckt

### M-04 ProfanityFilter — Trivial-Bypass via Leetspeak/Unicode
**Datei:** `Helpers/ProfanityFilter.cs:39-72`

- `Clean` operiert wort-fuer-wort mit case-insensitive, aber: `f*ck`, `fυck`
  (griechisches upsilon U+03C5), `fück`, `fuck.` (Punkt am Wortende wird zwar gestrippt,
  aber `f-u-c-k` mit Unterstrichen oder Zero-Width-Joiner nicht).
- Leetspeak (`fvck`, `nigg3r`) nicht abgedeckt.
- Diakritische Normalisierung fehlt (kein `String.Normalize(NormalizationForm.FormD)` plus
  `IsNonSpacingMark`-Filter).
- 6 Sprachen sind drin (DE, EN, ES, FR, IT, PT), aber sehr knappe Wortlisten (~6-12 pro
  Sprache). Play-Store-Compliance ist hier Basis-Schutz, aber bei Review kann das gerade
  noch durchgehen — nicht wenn die App in Tier-1-Markt gemeldet wird.

### M-05 Chat-Cooldown ist client-side
**Datei:** `Services/GuildChatService.cs:23-24, 34`

`_lastMessageSent` lebt im Client. Mit zwei Instanzen (z.B. Android+Desktop gleichzeitig)
schreibt der Spieler 12 Nachrichten/Minute statt 12. Rules-File haette via
`!data.exists() && ...` und `now - root.child('guild_chat/.../last_message').val()` ein

### M-06 GuildBoss-Belohnung: Preferences-Key teilt Boss-Identifier mit ALLEN Gilden
**Datei:** `Services/GuildBossService.cs:36, 477-501`

`PrefKeyLastBossRewardWeek = "guild_boss_reward_week"` ist ein einziger Key fuer den
gesamten Spieler. Spieler ist in zwei Gilden ueber den Zeitraum hinweg (verlaesst → joint),
die haben dieselbe Boss-StartedAt-Zeit (zufaellig, oder weil Boss-Spawn an Wochentag haengt)
— dann wird die zweite Gilden-Belohnung mit "lastClaimedBoss == bossIdentifier" geblockt
und der Spieler verliert die Reward.

### M-07 CoopOrder.Score-PATCH ohne Status-Lock
**Datei:** `Services/GuildCoopOrderService.cs:103-153`

`SubmitScoreAsync` patcht zuerst den Score (Zeile 124), re-readt, und patcht dann Status.
Wenn zwischen Re-Read und Status-PATCH der Status bereits von der Gegenseite auf `Expired`
gesetzt wurde (z.B. weil ExpiresAt zu kurz war), wird er hier wieder auf `Completed`
zurueckgesetzt — und beide Reward-Claimer holen sich den Reward, obwohl die Order eigentlich
abgelaufen war. Mitigation: PATCH mit `status: 2` (Completed) NUR wenn `data().status == 1`
(Active) — geht in REST nur mit Transactions (`POST .json?transaction=...`) oder via Rules.

### M-08 Boss-HP-Skalierung verliert Damage bei Member-Count-Aenderung
**Datei:** `Services/GuildBossService.cs:316-324`

`bossHp = baseBossHp * Math.Max(0.5, memberCount / 5.0)` wird beim Spawn berechnet. Wenn
zwischen Spawn und Defeat die `memberCount` sinkt (Member verlassen Gilde), bleibt die HP
auf dem alten Wert — die verbleibenden Spieler muessen mehr Damage liefern als per Capita
moeglich, weil die verlassenen Mitglieder ihren `guild_boss_damage`-Eintrag mitnehmen
(`LeaveGuildAsync:452` loescht sie). Boss faellt nie. Fair-Recovery: bei
`SpawnBossIfNeededAsync` retro-aktiv HP nachjustieren bei MemberCount-Drop. Aktuell fehlt.

### M-09 PushAsync re-postet nicht das gleiche `content` im 401-Retry
**Datei:** `Services/FirebaseService.cs:371-377`

`PostAsync` mit dem ORIGINAL `content` — `HttpContent` ist nach der ersten `PostAsync`-
Resolve aber konsumiert. Der zweite Aufruf sendet leeren Body → Firebase antwortet 400 oder
schreibt einen leeren Push-Key. PUT/PATCH-Pfade machen es korrekt mit neuer `StringContent`-
Instanz (Zeile 300-301), PushAsync nicht. Pop-Effekt: Token-Refresh-Race → Push-Verlust.

### M-10 SetOrderId ueber-feuert GameRestarted-Event bei Same-Order-Replay
**Datei:** `ViewModels/MiniGames/BaseMiniGameViewModel.cs:299-354`

Beim Drueckung von "Wiederholen" (Multi-Task-Order) wird `SetOrderId(sameId)` aufgerufen,
das setzt `IsPlaying=false; IsResultShown=false`, initialisiert das Game und feuert
`GameRestarted`. Wenn der Spieler in 100ms zweimal "Continue" drueckt (Reentrancy-Schutz
ist `if (!IsResultShown) return;` Zeile 633 — aber Continue setzt `IsResultShown` nirgendwo
explizit auf false, das macht erst SetOrderId), kann der Continue-Pfad zweimal navigieren.
Theoretischer Race; in Praxis unwahrscheinlich.

## Low / Hardening

### L-01 GuildChat liest 50 Nachrichten ohne Datum-Filter
**Datei:** `Services/GuildChatService.cs:95-99`

`orderBy="timestamp"&limitToLast=50` ist gut. Aber `timestamp` ist hier ein **String**
(ISO-O), und Firebase sortiert Strings lexikografisch — fuer ISO-O ist das gleichwertig
zu DateTime-Sort, aber wenn jemals jemand `timestamp` als `int` ms schreibt (z.B. via
`{".sv":"timestamp"}`), bricht die Sortierung. Konsistenz herstellen: ms-Timestamps
ueberall.

### L-02 InviteCode ohne Ablauf
Codes haben keine Expiry. Wenn eine Gilde 6 Monate alt ist, kursiert der Code potenziell
in alten Reddit-Threads. Mitigation: TTL via Rules + Client-Refresh.

### L-03 BrowseGuilds limitToLast=50 — Pagination fehlt
**Datei:** `Services/GuildService.cs:187-188`
Wachsende Gilden-Anzahl jenseits 50: Niedrig-Level-Gilden werden NIE gefunden. Solo-Spieler
findet seine Freunde nicht. Pagination per `endAt`/`startAt`-Cursor noetig.

### L-04 GuildResearchService — SetAsync statt UpdateAsync fuer einzelne Felder
**Datei:** `Services/GuildResearchService.cs:106, 251, 338`

Beim Abschluss eines Forschungs-Timers wird der ganze `GuildResearchState` per SetAsync
geschrieben — Last-Write-Wins gegen einen parallelen `ContributeToResearchAsync`. Wenn
Player A `Progress=400` schreibt und Player B im selben Moment `Completed=true`, kann
einer ueberschrieben werden. Atomare PATCH-Updates pro Feld waeren sicherer.

### L-05 `AvailablePlayerInfo.LastActive` String-Sort statt Numeric-Sort
**Datei:** `Services/GuildInviteService.cs:161`
`string.Compare(b.LastActive, a.LastActive, StringComparison.Ordinal)` — funktioniert
nur, weil ISO-O lex-sortierbar ist. Falls jemand das mal in MS-Timestamp wechselt: bricht.

### L-06 ProfanityFilter: keine Telemetrie ueber gefilterte Worte
Sinnvoll fuer A/B-Tuning der BlockedWords-Liste.

### L-07 `MaskFilter` (SkiaSharp) im MiniGame-Renderer nicht im Audit-Scope, aber CLAUDE.md
erwaehnt den OOM-Bug — relevant fuer Mini-Games mit Glow-Effekt (Forge!). Verifikation nicht
durchgefuehrt.

## Firebase Security Rules — Audit-Stichproben

### Validate-Coverage Matrix (Root-Rules)

| Pfad | indexOn | Monotonie | Score/Status-Lock | Author-Check | Rate-Limit |
|---|---|---|---|---|---|
| guilds/$gid | yes (level,leaguePoints,weeklyProgress) | yes (level +1) | no | guild-member | no |
| coopOrders/$oid | no | partial (createdBy/invitedPlayer immutable) | NO (Status frei) | guild-member NICHT player! | no |
| auctions/$aid | no | no | no | guild-member | no |
| megaProjects/$pid | no | yes (Contributions, Donations monoton) | yes (type/projectId immutable) | guild-member + donations gated | no |
| guild_chat/$gid | yes (timestamp) | n/a | n/a (msg immutable) | strict uid==auth | NO server-side |
| cloud_saves/$pid | no | no (savedAt frei!) | n/a | strict auth==player | n/a |
| analytics_events/$date/$push | no | n/a | n/a (immutable) | strict | no |
| guild_war_seasons (Root) | yes (status) | yes (points<= +10000, wins<= +10) | n/a | guild-member | n/a |
| guild_boss_damage | no | no (damage frei!) | n/a | $playerId==auth | no |

### Auffaelligkeiten
- **coopOrders Author-Check** fehlt → siehe C-04.
- **auctions Author-Check** fehlt → siehe C-05.
- **guild_boss_damage Monotonie** fehlt: ein Spieler kann sein eigenes `total_damage` von
  1.000.000 auf 5.000.000.000 setzen ohne Validierung → leaderboard-cheating.
- **cloud_saves.savedAt Monotonie** fehlt → siehe H-09.
- **megaProjects** ist die BESTE Rule-Struktur im File (Donations monoton, ProjectId immutable).
  Vorbild fuer die anderen.
- **App-lokale `database.rules.json`** ist ein RUECKSTAND zur Root-Variante — weniger Validatoren,
  keine `.indexOn` fuer Boss-/Achievement-Pfade. Welches deployed ist: unklar.

### Konsistente Probleme
- Kein einziger Pfad nutzt das `now`-Sentinel fuer Server-Timestamps oder Rate-Limits.
  CLAUDE.md verweist auf Firebase-Sentinel `{".sv":"timestamp"}` — wird im Code (geprueft
  via grep) **nicht verwendet**. Alle Timestamps client-gesetzt → keine Rate-Limits
  serverseitig.
- Es existiert kein zentrales `validate-immutability`-Pattern — manche Pfade haben
  `(!data.exists() || newData.val() == data.val())`, andere nicht.

## Zusammenfassung

Das Gilden-/Firebase-Subsystem hat eine durchdachte Client-Architektur (HMAC,
SemaphoreSlim, PATCH statt PUT, idempotente Claims, Master-Client-Pattern, PlayerId-
Stabilitaet ueber Account-Wechsel), aber die **Server-Rules sind nicht mit dieser
Verteidigungstiefe abgestimmt**: zwei kritische `.write`-Rules erlauben fremden
Gildenmitgliedern Lesen UND Schreiben auf fremde CoopOrder/Auctions (C-04, C-05), und
die Monotonie-Validierungen sind inkonsistent (megaProjects vorbildlich,
guild_boss_damage und cloud_saves komplett offen).

Die Cloud-Save-Konflikt-Erkennung (C-02) ist die groesste Spieler-UX-Falle —
Save-Verlust ohne Warning ist ein Tier-1-Bug, sobald mehr als ein Geraet im Spiel ist.
Im Mini-Game-Subsystem ist die Reentrancy-Absicherung (Continue-Doppel-Tap, GameRestarted-
Event-Order) solide bis auf Detailfaelle (M-02, M-10).

Empfehlung: Vor dem Production-Release der v2.1.0 Co-op-/Auction-Features mindestens
C-01 bis C-07 + H-02 + H-09 fixen. Die restlichen Findings sind Hardening-Aufgaben.
